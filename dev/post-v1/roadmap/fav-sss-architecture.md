# Favnir SSS (State-Stage-Sequence) Architecture

日付: 2026-05-01

## 概要

Favnir は、データ解析のライフサイクルを **SSS (State-Stage-Sequence)** の 3 つの概念に集約する。
これにより、誰が書いても同じ構造になる規約 (CoC) を徹底し、複雑なデータ変換を「明確な工程の連結」として再定義する。

---

## 1. State (状態): データの契約

データの「形」と、そのデータが満たすべき「不変条件」を定義する。

- **キーワード**: `type` (将来的に `state` も検討)
- **役割**: ビジネスルール（認証済、バリデーション済など）を型として表現する。
- **CoC**: 全てのデータは、その状態に応じた型を持たなければならない。

---

## 2. Stage (工程): データの変化

データを次の State へ進めるための、最小単位の処理工程。

- **キーワード**: **`stage`** (旧 `trf` / Transform)
- **役割**: 
    - 入力から出力への「一歩」の変換を記述する。
    - 副作用（Effect）を持つ場合、この段階で明示される。
- **CoC**: 1 つの `stage` は、単一の責任（Single Responsibility）を持つ工程として設計される。

```fav
stage NormalizeName: String -> String = |s| {
    String.trim(s) |> String.upper
}
```

---

## 3. Sequence (順序): データの構造

複数の Stage を繋ぎ合わせ、一連のデータ処理の流れ（パイプライン）を構築する。

- **キーワード**: **`seq`** (旧 `flw` / Flow)
- **役割**: 
    - `stage` の配線図。全体のストーリーを決定する。
    - ロジックを直接持たず、工程の連結に徹する。
- **CoC**: アプリケーションの全体像は、この `seq` を見るだけで（`fav explain` 等を通じて）把握できなければならない。

```fav
seq UserOnboarding = 
    NormalizeName 
    |> ValidateEmail 
    |> SaveUser
```

---

## 4. なぜ SSS なのか

| 概念 | 旧名称 | メンタルモデル | 役割 |
|:---|:---|:---|:---|
| **State** | Type | データの「静止画」 | 契約の保証 |
| **Stage** | Transform | データの「一歩」 | 工程の実装 |
| **Sequence** | Flow | データの「物語」 | 構造の定義 |

この 3 つに分けることで、手続き型言語にありがちな「だらだらした記述」は Stage の細分化によって抑制され、複雑な分岐は Sequence の階層化によって整理される。

---

## 5. v2.0.0 における構文の統合

以下のキーワードリネームは **v2.0.0 で確定的に実施する** 設計決定である：

| 現在 (v1.x) | v2.0.0 以降 | 理由 |
|:---|:---|:---|
| `trf` | `stage` | SSS の「Stage」に直接対応。意味が自明になる。 |
| `flw` | `seq` | SSS の「Sequence」に対応。`flow` より制御の順序を強調。 |
| `cap` | `interface` | 型への機能付与の意味が直感的になる。`abstract` との整合性も取れる。 |

v1.0.0 の Rust 製コンパイラは `trf` / `flw` / `cap` キーワードで動作するが、セルフホスト移行のタイミングで新構文へ一斉移行する。v1.x → v2.0.0 の移行ガイドを提供する。

---

## 一言でいうと

> State（状態）を決め、Stage（工程）を作り、Sequence（順序）で繋ぐ。
