# ADR-001: VM は Rust で実装する

- **状態**: 採用
- **決定日**: v4.x（初期設計時）
- **更新**: v9.0.0（セルフホスト完成時に再確認）

---

## 背景

Favnir はセルフホスト言語を目指しており、コンパイラ・型チェッカー・CLI を
Favnir 自身で実装することを目標にしてきた。
「VM も Favnir で書くべきか」という問いが v6〜v8 の開発中に生じた。

## 決定

VM（`fav/src/backend/vm.rs`）は恒久的に Rust で実装する。
Favnir で VM を書き直すことはしない。

## 理由

1. **メモリ安全性**: GC のないスタックマシンは Rust が最適。
   Favnir は現時点でポインタや低レベルメモリ操作を言語機能として持たない。

2. **自己参照の問題**: VM を Favnir で書くと、その Favnir を動かす VM が別途必要になる。
   無限後退を避けるには、どこかで Rust（または同等のホスト言語）に依存する必要がある。

3. **設計上の境界として有用**: 「VM は Rust」という境界が
   「Favnir でできること・できないこと」を明確にする。
   VM が提供する primitive を通じてのみ低レベル操作が可能、という設計が成立する。

4. **パフォーマンス最適化の余地**: 将来 JIT を追加するとき、Rust のままの方が現実的。

## 却下した選択肢

- **Favnir で VM を書く**: 上記の自己参照問題と、
  Favnir が低レベルメモリ操作を持たない設計矛盾で却下。
- **WebAssembly をホストにする**: 実行環境の依存が増える。
  WASM は Playground 向けターゲットとして別途対応済み（`favnir-wasm`）。

## 影響

- `fav/src/backend/vm.rs` は Rust のみで記述する
- 新機能の primitive は `vm.rs` の `call_builtin` match に追加し、
  Favnir 層（Rune）からラップする二層構造を維持する
- この決定は `INVARIANTS.md` II-1 に記載
