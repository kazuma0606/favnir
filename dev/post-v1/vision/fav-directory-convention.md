# Favnir Standard Directory Convention (SSS+V)

日付: 2026-05-01

## 概要

Favnir は、開発者がファイル構成に悩む時間をゼロにするため、厳格なディレクトリ規約 (CoC) を採用する。プロジェクトは **SSS+V (State-Stage-Sequence + Visual)** アーキテクチャに基づいて構成され、各ディレクトリの役割は物理的に分離される。

---

## 1. プロジェクト構造

全ての Favnir プロジェクトは、原則として以下の構造に従わなければならない。

```text
my_project/
├── fav.toml            # プロジェクトのメタデータ・テンプレート定義
├── migrations/         # SQL（DDL/DML）の唯一の保管場所。他での SQL 記述は禁止。
│   └── 001_init.sql
├── src/
│   ├── states/         # [State] 型定義、不変条件、スキーマ定義
│   ├── stages/         # [Stage] 原子的な工程（ロジックの実装）
│   ├── seqs/           # [Sequence] 工程の配線・パイプライン構造
│   ├── visuals/        # [Visual] 可視化、ノートブックのレイアウト定義
│   └── main.fav        # エントリポイント
├── tests/              # 各レイヤー（State, Stage, Seq）に対応したテスト
└── fav.lock            # 依存関係のロックファイル
```

---

## 2. 各ディレクトリの規約と制限

### `migrations/` (The SQL Sandbox)
- **役割**: DB スキーマの変更（DDL）や初期データの投入（DML）を行う。
- **制約**: `src/` 配下で SQL 文字列をハードコードすることは厳禁。DB 操作は `states/` の型定義から導出された Job インターフェースを通じてのみ行う。

### `src/states/` (The Contract)
- **役割**: `type` と `invariant` を用いて、データの「形」と「契約」を定義する。
- **制約**: 実行可能なロジックや副作用（Effect）を持つ関数をここに書いてはならない。

### `src/stages/` (The Implementation)
- **役割**: `stage`（旧 `trf`）を用いて、データの変換や外部連携の「一歩」を実装する。
- **制約**: 1 つの `stage` に複数の責任を持たせない。複雑なロジックは細分化し、上位の `seq` で連結する。

### `src/seqs/` (The Structure)
- **役割**: `seq`（旧 `flw`）を用いて、`stage` をパイプラインとして連結する。
- **制約**: ここに具体的な計算ロジックや `if` / `match` による複雑な分岐を直接書くことは禁止。分岐が必要な場合は、その判断自体を `stage` として切り出す。

### `src/visuals/` (The Presentation)
- **役割**: ノートブック（Veltra）でのデータの見せ方（チャート、テーブル、色のマッピング、配置）を定義する。
- **目的**: 伝統的な View (UI) ではなく、**Visualisation (可視化)** に特化する。
- **制約**: ビジネスロジックをここに書いてはならない。

---

## 3. なぜ「決め打ち」なのか

1. **意思決定の排除**: 「どのディレクトリにファイルを作るか」という無駄な思考を排除する。
2. **自動化の加速**: 構造が固定されているため、`fav build` や `fav explain` が常に正確なメタデータを抽出・生成できる。
3. **保守性の向上**: 他人が書いたコードであっても、ディレクトリを見た瞬間にどこに何があるか（State なのか Sequence なのか）が即座に理解できる。
