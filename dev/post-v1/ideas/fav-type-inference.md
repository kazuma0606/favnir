# Favnir Type Inference & Human-in-the-Loop Design

日付: 2026-05-02

## 概要

Favnir の型推論は「暗黙の魔法」ではなく、**「開発者（人間/AI）とコンパイラの対話」** を円滑にするための道具として設計する。

既存の関数型言語（Haskell 等）のような強力なグローバル型推論は、コードの可読性（暗黙的な型の変化）や AI によるコード生成の制御を困難にする可能性がある。Favnir では **「境界の明示性」** と **「局所的な推論」** を組み合わせたアプローチをとる。

---

## 1. 境界の明示性 (Explicit Public Boundaries)

`fn`, `stage`, `seq` といったトップレベルの構成要素（SSS）のシグネチャは、原則として**省略不可**とする。

```fav
-- 良い例: 境界が明確
stage ProcessData: RawData -> ValidatedData! = |data| { ... }

-- 悪い例（検討中）: 戻り値が推論される
-- stage ProcessData: RawData -> _ = |data| { ... }
```

### なぜ明示するのか
1.  **契約 (CoC)**: 境界の型はコンポーネント間の契約である。契約が暗黙的だと、内部実装の変更が予期せぬ契約変更を引き起こす。
2.  **AI へのアンカー**: AI がコードを生成する際、明示的な型は「守るべきゴール」として機能し、生成の精度を向上させる。
3.  **人間によるレビュー**: `fav explain` で見たときに、入出力が常に一目でわかる。

---

## 2. 局所的な型推論 (Local Inference)

関数やステージの内部（`bind`, `chain`）では、右辺の式から型を自動的に推論する。これは既に実装済みだが、さらに強化する。

```fav
fn example() -> Unit {
    bind n <- 10              -- n: Int (推論)
    chain user <- fetch_user  -- user: User (Result からの推論)
}
```

---

## 3. 型ホール (Type Holes) と Human-in-the-Loop

「型は書きたいが、複雑なジェネリクスなどを手で書くのが面倒」という場合のために、**型ホール (`_`)** を導入する。

```fav
bind users: _ <- Db.query("SELECT ...")
```

### 動作モデル
1.  開発者が `_` を書く。
2.  LSP (Favnir Language Server) がその位置にある型を推論する。
3.  **Human in the loop**:
    - LSP が推論結果をエディタ上にインレイヒントとして表示。
    - 開発者（または AI ツール）が「型の固定」コマンドを呼ぶと、`_` が具体的な型名（例: `List<UserRow>`）に置換される。
4.  これにより、「手抜きはできるが、最終的なコードには型が残る」という状態を作る。

---

## 4. パイプライン推論 (Pipeline Flow Inference)

`|>` 演算子による連結では、前の工程の出力が次の工程の入力に一致することをコンパイラが保証する。この連結部分での型指定は不要。

```fav
data
|> StageA  -- T1 -> T2
|> StageB  -- T2 -> T3
|> StageC  -- T3 -> T4
```

この流れにおいて、中間状態 `T2`, `T3` をユーザーが意識する必要はない（`fav explain` では確認可能）。

---

## 5. 将来の拡張：双方向型検査 (Bidirectional Type Checking)

より高度な推論（ジェネリクスの具体化など）を安定して行うため、コンパイラ内部に「期待される型を下に伝える (Checking)」と「式から型を取り出す (Synthesis)」の双方向アプローチを導入する。

### 例：ラムダ式の推論
```fav
fn map_example(xs: List<Int>) -> List<String> {
    xs |> List.map(|x| x.to_string())
    -- List.map の第1引数が Int -> String であることを期待しているため、
    -- 引数 x の型を Int と推論できる。
}
```

---

## 一言でいうと

> 「推論」は型を書くための**補助**であり、型を**隠す**ための機能ではない。
> AI と人間が共に「正しさ」を確信できるコードを目指す。
