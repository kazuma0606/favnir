# Favnir Manifesto: Why Favnir?

日付: 2026-05-01

## 概要: 既存の枠を越える

Favnir は、既存のプログラミング言語が「計算の効率」や「メモリの安全」に注力してきたのに対し、**「データの正しさ（信頼性）」** と **「構造の説明可能性（透明性）」** に全リソースを投じる。

Rust や Python が存在する現代において、なぜ Favnir が必要なのか。その答えは、データ解析の現場における「自由すぎるがゆえの混乱」を終わらせるためである。

---

## 1. メモリ安全から「データ安全」へ

Rust はメモリ管理のミスを排除したが、ビジネスルール（データ）のミスを排除する仕組みは標準化されていない。

- **既存の課題**: 「未認証のユーザーが削除処理を呼べてしまう」「負の数の年齢が DB に登録される」といった論理的バグ。
- **Favnir の解決**: `State` (型) と `Invariant` (不変条件) の統合。
    - **Semantic Types**: `PositiveInt` や `Email` といった「現実世界の状態」を標準ライブラリ (`std.states`) として提供。
    - 開発者はバリデーションを書くのではなく、適切な **State（型）を選択するだけ**で、データの正しさをシステム全体で保証する。

---

## 2. 再利用性から「説明可能性」へ

高度に抽象化されたコードは再利用には優れるが、非エンジニアや監査担当者にとってはブラックボックスである。

- **既存の課題**: 「この計算結果はどうやって導き出されたのか？」を追うために、何百ものファイルを辿る必要がある。
- **Favnir の解決**: パイプライン（Sequence）の第一級市民化。
    - `fav explain` や `fav graph` により、データの流れが常に「見える化」される。
    - 抽象化（interface, abstract seq）を行っても、その「配線図」は常に明示的であり、隠蔽ではなく構造化として機能する。

---

## 3. 万能性から「最強の制約（CoC）」へ

「何でもできる」ことは、大規模・長期的なプロジェクトでは負債になる。

- **既存の課題**: プロジェクトごとにバラバラなディレクトリ構造、エラーハンドリング、DB 接続の作法。
- **Favnir の解決**: 徹底した **Convention over Configuration (CoC)**。
    - **SSS+V Layout**: ディレクトリ構造の完全固定。
    - **SQL 隔離**: ソースコードからの SQL 排除と Job 投入型アーキテクチャ。
    - 開発者は「どう設計するか」に悩む必要がなく、レールの上の「ロジックの実装」だけに集中できる。

---

## 4. ポジショニング： Rust vs Favnir

| 観点 | Rust | Favnir |
|:---|:---|:---|
| **関心の中心** | コンピュータ（メモリ、CPU） | データ（状態、契約） |
| **安全性の対象** | 不正なメモリ操作の排除 | 不正なデータ状態の排除 |
| **抽象化の方向** | 高度な再利用性 | 高度な説明可能性 |
| **自由度** | 極めて高い | 規約による強い制約 |
| **主な用途** | システム、基盤、高負荷処理 | データ解析、ETL、ビジネスロジック |

---

## 結論： Favnir が目指す世界

Favnir は、プログラミングを「命令の羅列」から「正しいデータの物語（Sequence）」へと昇華させる。

エンジニアが想像力を発揮すべきは「どうやって動かすか」という低レベルな詳細ではなく、**「データがどうあるべきか（State）」** と **「データがどう変化すべきか（Stage）」** という高次の設計である。

> **「Rust はコンピュータに寄り添うが、Favnir はデータに寄り添う。」**
