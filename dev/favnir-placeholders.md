# Favnir Placeholder Draft

更新日: 2026-04-26

## 目的

このメモは、Favnir におけるプレイスホルダー構文の最小仕様を整理するための草案。

プレイスホルダーは、高階関数や pipeline を読みやすくする一方で、曖昧さも増やしやすい。  
そのため、最初はかなり狭く定義した方がよい。

## 基本方針

最初に入れるなら、単一プレイスホルダー `_` に絞る。

狙い:

- 小さいクロージャの冗長さを減らす
- `map`, `filter`, `group` の読みやすさを上げる
- 関数型言語らしい簡潔さを出す

ただし、

- 複数引数
- 暗黙の優先順位
- 複雑な式の推論

までは最初から広げない。

## 許可したい形

### 1. field access

```fav
rows |> map(_.email)
users |> group(_.department)
```

これは最優先で入れたい。

理由:

- 直感的
- よく使う
- 型推論しやすい

意味:

```fav
|x| x.email
```

の sugar。

### 2. method-like access

```fav
rows |> filter(_.is_active())
```

これも許可してよい。

意味:

```fav
|x| x.is_active()
```

### 3. 単純な式

```fav
numbers |> map(_ + 1)
numbers |> filter(_ > 0)
```

これもかなり有用。

意味:

```fav
|x| x + 1
|x| x > 0
```

ただし、最初は「単一の `_` のみを含む式」に制限した方がよい。

## 最初は後回しにしたい形

### 1. 複数プレイスホルダー

```fav
pairs |> map(_1 + _2)
```

または

```fav
fold(items, 0, _ + _)
```

これは便利だが、最初は避けたい。

理由:

- どの `_` が何を指すか曖昧になりやすい
- parser / checker が重くなる
- AI も誤解しやすい

### 2. `$0`, `$1` 形式

```fav
map($0.email)
```

最初は不要。

理由:

- 記号として読みにくい
- 関数型らしさはあるが、Favnir の顔としてはあまり美しくない

### 3. ネストが深い式

```fav
rows |> map(_.profile.address.city.upper())
```

これは許可してもよいが、型エラー時の診断が難しくなるので、最初は段階的導入でもよい。

## `_` の意味

プレイスホルダー `_` は、「その文脈で期待される単一引数関数の引数」を表す。

例:

```fav
rows |> map(_.email)
```

ここでは `map` が `T -> U` を要求するので、`_` はその `T` に対応する。

つまり:

```fav
_.email
```

は単独で意味を持つのではなく、関数を期待する文脈の中で

```fav
|x| x.email
```

へ展開される。

## 許可文脈

最初は `_` の使用文脈を絞る。

許可候補:

- `map(...)`
- `filter(...)`
- `group(...)`
- `flat_map(...)`
- `sort_by(...)`

あるいは一般化して、

- `A -> B` を期待する引数位置

に限る。

自分のおすすめは後者。  
つまり、プレイスホルダーはライブラリ個別対応ではなく、型で一般化した方がよい。

## `|x| ...` との関係

プレイスホルダーはクロージャの完全な代替ではない。

原則:

- 短い単引数変換は `_`
- 複雑な処理は `|x| ...`

例:

```fav
rows |> map(_.email)
rows |> map(|row| normalize_email(row.email))
```

この使い分けが自然。

## `stage` との関係

プレイスホルダーは主に高階関数用であり、`stage` 定義そのものを置き換えるものではない。

```fav
stage NormalizeEmail: User -> User = |user| {
    ...
}
```

これは名前付き再利用単位。  
一方、

```fav
rows |> map(_.email)
```

はその場限りの小さい変換。

## parser / checker 観点

最初に制限を強くすべき理由はここにある。

最小ルール:

- `_` は単一引数クロージャへ展開される
- 一つの placeholder 式には単一の `_` だけ
- `_` は関数を期待する文脈でのみ許可

これなら parser / checker ともかなり扱いやすい。

## 推奨初期仕様

最初に許可するのは次。

- `_.field`
- `_.method(...)`
- `_ + 1`
- `_ > 0`
- `_ == value`

最初は入れないもの:

- `_ + _`
- `_1`, `_2`
- `$0`, `$1`
- 複数引数 placeholder
- 文脈なしの `_`

## 具体例

### 良い例

```fav
rows |> map(_.email)
rows |> filter(_.active)
users |> group(_.department)
numbers |> map(_ + 1)
numbers |> filter(_ > 10)
```

### 後回しにする例

```fav
pairs |> map(_1 + _2)
items |> fold(0, _ + _)
map(_.a + _.b)
```

## 仮の結論

Favnir のプレイスホルダーは、最初はかなり小さく始めるのがよい。

- 単一プレイスホルダー `_`
- 単一引数関数文脈のみ
- `_.field` と単純な式を優先

これで読みやすさはかなり上がる。  
複数プレイスホルダーや高度な省略記法は、その後に必要性を見て拡張すればよい。
